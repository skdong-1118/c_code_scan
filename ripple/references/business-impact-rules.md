# 业务影响发现规则

本文件用于帮助模型从代码变更出发，发现开发者可能没有意识到的业务影响流程。

## 核心目标

分析必须回答：

- 这次修改改变了什么业务行为？
- 哪些业务入口会走到这段代码？
- 哪些其他功能复用了同一段公共流程？
- 哪些条件、状态、模式或消息类型会触发不同路径？
- 哪些老功能、旁路流程或间接入口容易被忽略？

## 变更语义

对每个 changed subject，先说明它改变了哪类行为：

- 输入如何被解释、过滤或校验
- 状态如何转换
- 请求、消息或事件如何分发
- 配置如何读取、应用或回滚
- 数据如何生成、修改、缓存、排队或发送
- 返回结果、副作用或通知行为如何变化
- 业务条件、模式、类型、标志位或策略如何变化

不要只复述代码差异。必须把代码变化翻译成业务语义。

## 业务入口

沿调用链向上查找所有可能入口，包括：

- 北向 API 或管理接口
- CLI / command
- 配置加载、配置下发、查询或同步入口
- 消息接收与 dispatch
- 事件、定时器、状态机入口
- callback / handler / ops table 触发入口
- 初始化、恢复、升级或兼容流程
- legacy / stable / old feature path

入口名字不明显时，根据 caller、文件路径、消息类型和相邻逻辑解释其业务含义。

## 共享公共流程

如果 changed function 是公共函数、底层 helper 或通用处理流程，必须继续向上展开：

- 直接调用它的业务流程
- 多层 wrapper 之后才分叉的业务流程
- 跨 subsystem 的复用路径
- 同一入口下不同 mode/state/type/flag 对应的流程

不要只列最近一层 caller。目标是识别所有具有不同业务含义的上游路径。

## 容易忽略的影响

重点寻找：

- 函数名看起来无关，但实际复用了 changed flow 的调用方
- 老功能或兼容路径
- 异常输入、空数据、边界状态、恢复流程
- 只有特定配置、模式、消息类型或状态下才触发的分支
- 注册式、表驱动或间接 dispatch 路径
- 一个底层流程同时服务多个 subsystem 的情况
- changed function 的副作用被其他业务读取或消费的情况

## 业务影响证据

每条候选影响流程必须记录：

- 业务入口
- 完整或尽可能完整的调用路径
- 触发条件
- changed subject 在该流程中的作用
- 可能改变的业务结果
- 证据状态

证据状态使用：

```text
confirmed：调用关系和源码语义均能确认
probable：存在完整调用路径，但业务条件仍需验证
possible：存在引用或共享数据关系，路径尚未闭合
evidence_gap：证据不足，不能排除影响
```

## 完成标准

业务影响分析只有在满足以下条件后才算充分：

- changed subject 都已映射到业务语义
- 主要上游业务入口已展开
- 共享公共流程的不同业务分支已区分
- 容易忽略的流程已单独检查
- 未闭合路径已明确记录为 evidence gap
- 验证建议与具体业务场景对应
